# 公众号内容打分

两种模式，按用户问的来：

| 模式 | 评估对象 | 何时用 | 框架 |
|---|---|---|---|
| **A. 标题+封面打分** | 只评标题+封面在信息流的吸引力 | 写完正文、做封面之前；封面做完、发布前；历史复盘 | 七维 100 分制（见下文） |
| **B. 全文打分** | 评整篇文章的传播力、阅读体验、认知价值 | 写完正文后看整体内容质量；发布前最终质检；写完一批文章对比 | 资深主编版八维 100 分制（见下文） |

**关键原则**：
- 模式 A 不评正文（用户明确说不要）
- 模式 B 评全文但**不评观点对错**（用户明确说不要）——只评"作为公众号内容的传播力"
- 用户没说哪种时，默认按"标题+封面打分"（模式 A）

---

## 模式 A：标题 + 封面流量潜力打分

只评估**标题 + 封面**在公众号信息流里的吸引力，不评正文质量。

## 适用场景

- 写完文章准备发之前，给标题和封面打分
- 看别人文章时复盘为什么火/为什么没人看
- 同一篇文章尝试多个标题时挑最优

## 评分框架（总分 115）—— 真实点击潜力 + 真实性 + 账号风格版

**核心原则**：评的是"目标读者在手机信息流里**会不会点开**"+"点了不失望"+"不破坏账号长期品牌"，不是"标题写得是否工整"。

**绝对禁止的隐性偏差**：
- ❌ 不要因为标题有"我以为/后来发现/帮我/时序"就自动给高分
- ❌ 不要因为标题结构完整（弧线、对比、时序）就给高分
- ❌ 不要把"抽象高级"误判成"高点击"
- ❌ 不要为了"短期流量"牺牲"账号长期品牌"和"读者信任"
- ✅ 结构完整性只能是**加分项**，不能替代**热词/身份代入/1秒理解/具体信息承诺/真实性/账号风格**

**三段式评分**：

| 段 | 维度 | 分值 |
|---|---|---|
| **一、七维流量潜力** | 真实点击潜力 | 100 |
| **二、真实性与预期一致性** | 标题承诺与正文兑现 | 15 |
| **三、账号风格匹配度** | 长期品牌一致性 | 10 |
| **总** | | **125** |

### 第一段：七维流量潜力（100 分）

| 维度 | 分值 | 核心问题 | 评分锚点（举例） |
|---|---|---|---|
| 1. 目标读者代入感 | 20 | 读者一眼觉得"这是在说我"？ | 有明确身份词/场景词/工具词：程序员、Claude Code、Cursor、Codex、AI创业、产品经理、OpenClaw、vibe coding、Agent |
| 2. 热词与讨论场接入 | 15 | 是否接入目标读者最近正在讨论的话题？ | 是否包含最近 AI/产品/创业圈高频词（vibe coding、Claude Code、Cursor、Codex、Agent、Skill、MCP、AI创业），是否能让读者感觉"最近正在发生" |
| 3. 1秒理解成本 | 15 | 读者 0.3 秒能不能立刻知道在讲什么？ | 标题能不能在第一眼就传达"这篇文章解决什么"；是否需要读者思考两遍才懂；是否过于抽象（"淘汰错误想法"这种表达要判断是否清楚） |
| 4. 认知冲突/反直觉 | 20 | 是否有"我以为A，结果B"或"大家以为A，我看到B"？ | 反转/反问/反常识；是否能制造"为什么会这样"的好奇 |
| 5. 信息承诺清晰度 | 15 | 点进去能获得什么是否明确？ | 是经验、复盘、判断、方法论，还是泛泛观点；是否避免空泛大词（"AI时代""未来趋势"） |
| 6. 传播/讨论欲 | 10 | 是否容易让人想转给同事/朋友？ | 能否引发认同/争论/站队/"这说的是我"；有无金句潜力 |
| 7. 语言锋利度与节奏 | 5 | 是否顺口、短、利，有记忆点？ | 是否避免太长太绕；有无"动词钩子"（淘汰/做对/重做）；有无对仗/押韵 |

**注**：封面视觉焦点、标题封面一致性 → 移到 Step 0 单独预检（封面打分），不挤进标题七维。

### 第二段：真实性与预期一致性（15 分）

**这是"防止短期流量自杀长期品牌"的核心维度**——读者点开是第一步，读完不失望并继续关注是第二步。

**评估清单**：

1. 标题承诺的内容，正文是否真的提供
2. 标题中的数字是否有真实来源（正文数据 / 作者亲历 / 行业可查）
3. 标题是否会让读者形成错误预期
4. 标题是否存在"点击率高但读完失望"的风险

**加分项**：
- + 标题中的数字来自作者真实经验
- + 标题中的数字来自正文明确数据
- + 标题与正文高度一致
- + 读者读完不会产生"被骗进来"的感觉

**扣分项**：
- - 数字看起来像行业统计，但正文无法支撑（如"90% 程序员都不知道"，正文没有数据来源）
- - 标题承诺的内容正文没有回答
- - 为了点击率故意放大结论
- - 读者点进来期待 A，结果正文讲 B

**评分标准**：

| 分数 | 含义 | 典型症状 |
|---|---|---|
| 13-15 | 标题与正文高度一致，无明显误导 | 数字有来源 / 承诺的具体 / 读完符合预期 |
| 10-12 | 存在轻微包装，但整体符合预期 | 标题有"放大"但正文撑得住 |
| 6-9 | 存在明显预期偏差 | 数字悬空 / 承诺的内容只在边缘提到 |
| 0-5 | 严重标题党风险 | 数字编造 / 内容与承诺完全不符 |

### 第三段：账号风格匹配度（10 分）

**这是"防止短期流量破坏长期品牌"的核心维度**——某些标题短期能涨粉 1000，但风格不符会让老读者觉得"号变了"，长期掉粉更严重。

**作者特点**（必须匹配）：
- AI 实践
- OpenClaw
- Claude Code
- 产品实验
- 软件工程
- 系统思维
- 反馈循环
- 复杂性管理

**作者不是**（必须避免）：
- 情绪博主
- 焦虑博主
- 数字党
- 标题党

**加分项**：
- + 真实经验
- + 方法论提炼
- + 反直觉观察
- + 第一手实践
- + 系统思维暗示（"反馈循环""复杂性"）

**扣分项**：
- - 过度夸张
- - 强情绪刺激
- - 制造焦虑
- - 数据党标题（"90% 的人都不知道"）
- - 震惊体
- - 鸡汤化

**评分标准**：

| 分数 | 含义 | 典型症状 |
|---|---|---|
| 8-10 | 非常符合账号长期品牌 | 真实经验 / 方法论 / 第一手实践 |
| 5-7 | 短期流量可能好，但不利于品牌积累 | 略夸张 / 略情绪化 |
| 0-4 | 明显偏离账号定位 | 数字党 / 震惊体 / 焦虑 / 标题党 |

## 扣分项（独立于七维，可叠加）

**重要：扣分项要严于加分项**——很多标题"没扣分但也没真的给读者留下印象"，这种其实是低潜力。

| 扣分症状 | 扣分 | 典型案例 |
|---|---|---|
| **直接给结论但没有问题感** | -5 ~ -15 | "AI 时代产品方法论彻底变了"（直接宣布观点）|
| **抽象词太多，缺少具体对象** | -5 ~ -20 | "做产品实验"（"实验"太抽象）/ "AI 时代的本质"（"本质"空）|
| **太像行业评论/AI 泛观点** | -5 ~ -15 | "我看到了 AI 创业的真相" / "未来已来" |
| **读者无法判断和自己有什么关系** | -10 ~ -20 | "我们都在被 AI 重塑"（"我们"太宽泛）|
| **虽然结构完整但点击欲弱** | -5 ~ -15 | 完整"我以为A结果B"弧线但缺少热词/身份词 |
| **字数 > 28 且信息密度不够** | -5 ~ -10 | 32 字的口语化标题（前 22 字铺垫，钩子被截断）|
| **关键词太宽泛，没有具体锚点** | -5 ~ -15 | "AI 时代""未来趋势""大模型""创业"（缺具体工具/场景）|

## 工作流

### Step 0：标题长度预检（必走，独立于七维）

公众号信息流里，标题的物理显示宽度有限。
- 手机端单行约显示 15-20 字（中文按 1 字算，英文按 1 字符算 1 字，数字按 1 字算）
- 超过 22 字必被截断，钩子失效
- 短于 11 字，身份词/反转/冲突塞不进去
- **15-20 字是最佳区间**（既能在缩略图完整显示，又能塞下"身份词 + 反转"）

**字数扣分表**（从总分里直接扣，不挤进七维）：

| 标题字数 | 扣分 | 原因 |
|---|---|---|
| 15-20 字 | 0 | 最佳区间，单行完整显示+信息密度够 |
| 13-14 或 21-22 | -1 | 边界外，可接受但欠优 |
| 11-12 或 23-24 | -2 | 明显偏离，信息密度或显示完整度不足 |
| 9-10 或 25-26 | -3 | 严重偏离，要么太空要么必截断 |
| <9 或 ≥27 | -4 | 极短（丢失冲突/身份词）或必截断（前缀钩子丢失） |

**注**：
- "字" = 中文字符、英文字母按 1 字符、数字按 1 字符、英文空格不计入
- 例外：纯英文标题（无中文）按 1 词 = 1.5 字折算

### Step 0.5：标题-正文一致性预检（必走，独立于七维）

**这是公众号最隐蔽也最致命的扣分项**——读完落差让读者觉得"被骗了"，直接打击转发欲（往往是低传播率的根因）。

**判断方法**：
1. 标题里的核心关键词（如 "Claude Code"）点出来的是什么视角？—— 工具视角 / 方法论视角 / 趋势视角
2. 正文实际在讲什么视角？—— 用一句话总结正文的真正核心观点
3. **两者是否同一视角？**

**扣分表**（从总分里直接扣，不挤进七维）：

| 错位程度 | 扣分 | 典型症状 |
|---|---|---|
| 完全一致 | 0 | 标题承诺的"会得到什么"和正文实际给的"是什么"对得上 |
| 视角错位 | -3 | 标题说"工具"，正文讲"方法论"；或标题说"个人经验"，正文讲"行业判断"（典型：工程师点进"Claude Code 半年"标题，结果读到产品哲学） |
| 范围错位 | -5 | 标题限定一个具体场景（如"做八字 Skill"），正文泛化到整个产品方法论（读者期望八卦/具体复盘，结果读到一篇大而泛的方法论文） |
| 承诺完全不符 | -8 | 标题说"A 体验"，正文讲"B 主题"（这种基本是不小心的写作事故） |

**典型错位举例**（要记住，下次自动避坑）：

❌ 标题："Claude Code 用半年我才发现：它做的是产品实验"
✅ 正文实际：讲的是 AI 时代产品方法论（"做对→做快"）
❌ 错位：标题锁定工具视角（"它做的是 X"），正文实际是方法论视角
❌ 后果：工程师点进来看 Claude Code 技巧 → 读完发现是产品哲学 → 觉得"被骗了" → 不转发
**应改成**：
- 改成方法论视角："Claude Code 用半年我才发现：做产品方法论变了" / "Claude Code 半年后：AI 产品方法论从'做对'变成'做快'" / "Claude Code 用半年：让我重新理解'做对'和'做快'"
- 或者改成工具视角的真正具体经验（不写方法论）

**修法检查清单**：
- 改完标题后，把标题关键词列出来
- 再用一句话总结正文核心观点
- 两者视角必须一致（要么都"工具"，要么都"方法论"，要么都"个人经验"）
- 不能一个限定"工具"另一个讲"方法论"

### Step 0.6：标题拗口度预检（必走，独立于七维）

**拗口 = 重复读 = 信息流划走**。公众号信息流里读者读标题 0.3 秒，**任何需要"重新组句才能懂"的标题就是拗口**。

**判断方法（逐条检查）**：

1. **双引号/书名号数量**：标题里出现 2 个及以上"X"或《X》→ 大概率拗口（每出现一组 +1 拗口分）
2. **主语跳跃**：标题前半部分在讲 A，后半部分突然转 B，中间没有连接词 → 拗口
3. **"X 不如 Y" 结构**："X"和"Y"必须都是读者已知的明确概念，否则读者会卡在"X是什么"——拗口
4. **嵌套结构**："A，B，C，D"超过 3 个短句、或"X，Y"出现 2 次以上 → 拗口
5. **绕口令测试**：标题一口气读完是否要中途换气 2 次以上？要换气 2 次 = 拗口

**拗口扣分表**（从总分里直接扣，可叠加，最多 -8）：

| 拗口症状 | 扣分 | 典型案例 |
|---|---|---|
| 0 个拗口症状 | 0 | 流畅，一口气读完 |
| 1 个轻微症状（如 1 对引号 / 1 次主语跳跃） | -2 | 还能读，但不顺 |
| 2 个症状（如 2 对引号 + 1 次主语跳跃） | -4 | 明显拗口 |
| 3 个症状（如 2 对引号 + 主语跳跃 + "X 不如 Y" 都陌生） | -6 | 必须重读 |
| 4 个及以上 | -8 | 灾难级拗口 |

**改法原则**：
- **去掉引号**：除非你引的是大众熟悉的金句（"996" "35 岁"），否则一律去掉
- **主语统一**：标题全程围绕同一个主语，不要从 A 跳到 B
- **"X 不如 Y" 的 X 和 Y 必须是大众词**：X=快、便宜、稳（已知概念）；Y=慢、贵、玄（已知概念）。如果 X 或 Y 是生造词，必须改写成大众词
- **短句重排**：如果标题有 3 个以上短句，砍到 2 个短句（逗号最多 1 个）

**改法举例**：

❌ `Claude Code 半年，我才看懂"做对"不如"做快"`
拗口：2 对引号 + 主语跳跃（Claude Code → 做对 → 做快）+ "X 不如 Y" 双方都是生造短语
✅ 改成 `Claude Code 半年，我才看懂一件事：做对不如做快`（去掉引号，主语统一为"一件事"，X/Y 简化为大众词）
✅ 或 `Claude Code 半年，让我重新理解了一件事`（彻底规避"X 不如 Y"结构）
✅ 或 `Claude Code 半年，我才看懂"做对"是个错觉`（保留 1 对引号，但 X 是大众词，Y=错觉也是大众词）

### Step 1：接收用户输入

用户需要提供：
- 【目标读者】（必填，例：程序员 / AI工程师 / OpenClaw用户 / AI创业者）
- 【标题】（必填，**可以同时粘贴多个标题对比**）
- 【封面】（必填，图片路径或URL；如果只评标题不评封面，封面那部分标 N/A）
- 【已知参考数据】（可选，历史阅读/点赞/分享数据）
- 【文章核心观点】（强烈建议填——评分时需要用核心观点判断"信息承诺"是否兑现）

如果只提供标题没有封面：明确告知"封面维度无法评估"，其余六维照打。
如果提供多个标题：分别打分，最后输出"标题选择建议"（按真实点击潜力排序）。

### Step 2：评估封面

封面评估**必须调用 vision 工具实际看图**，不要凭空判断。

**方法 A（推荐）**：用 `mcp_browser_navigate` 打开 file:// 本地路径，再 `mcp_browser_vision` 出图分析。
**方法 B**：`mcp_vision_analyze` 直接传本地路径（外链URL常失败）。

**评估清单**（缩略图视角，假设手机端约 200x150 像素）：

| 检查项 | 红旗信号 |
|---|---|
| 视觉中心 | 没有明确主体，或有2个以上并列主体 |
| 主文字字号 | <画面15%面积，或多行小字 |
| 元素数量 | 主体+文字+装饰>4个独立元素 |
| 配色对比 | 主文字与背景亮度差<40% |
| 缩略图可读性 | 缩到200x150后主体仍然清晰 |
| 视觉层级 | 没有"主标题文字>主体图像>背景"三段式 |

### Step 3：标题七维打分

每个维度**必须给具体扣分理由**，不能直接给总分。

#### 1. 目标读者代入感（/20）

**加分词**（命中任意+3~5分）：程序员 / 软件工程 / vibe coding / Claude Code / Cursor / Codex / Agent / OpenClaw / AI创业 / 副业 / 35岁 / 转型 / 失业 / 产品经理
**减分信号**：
- 标题里完全没有身份词/场景词（"我们""你"等代词不算）
- 标题用了"我们"这种宽泛代词，没有具体身份锚点

#### 2. 热词与讨论场接入（/15）

**加分**：标题包含最近3个月目标读者高频讨论的关键词（Claude Code / Cursor / Codex / Agent / Vibe Coding / Prompt / 上下文工程 / Skills / MCP / OpenClaw 等）
**减分**：
- 完全泛AI观点（"AI时代来临"），无具体产品/技术/事件挂钩
- 关键词太宽泛（"大模型""创业""未来"），无具体锚点（-5~-15）

#### 3. 1秒理解成本（/15）

**高分锚点**：
- 标题第一眼就能传达"这篇文章在解决什么"
- 动词具体（淘汰/重做/做错/验证）
- 抽象词少（避免"本质""价值""范式""复杂性"等空词）

**低分信号**：
- 需要思考两遍才懂
- 抽象词堆砌（"AI时代的本质"）
- 主语跳跃（读者要重新组句）

#### 4. 认知冲突/反直觉（/20）

**高分量表达**：
- "我以为A，结果B"（"以为…其实…"）
- 反问句（"为什么…？"）
- 反转（"不是X，是Y"）
- 数字反差（"92%技术评估准确率，只有11%需求命中"）
- 极值词（"最危险的""真正稀缺"）
- 强动词（淘汰/重做/颠覆）

**减分**：标题是"AI时代要X""程序员应该Y"这种无冲突的判断句

#### 5. 信息承诺清晰度（/15）

**加分**：读者能预知文章会给出"具体复盘/具体判断/具体数字"
**减分**：
- 标题像口号（"未来已来""新范式"）
- 标题太抽象（"复杂性""价值"等抽象名词堆砌）
- 标题没有承诺任何"读者会得到的东西"

**注**：如果用户提供了【文章核心观点】，要交叉验证——标题承诺的"会得到什么"和文章实际给的"是什么"是否对得上。

#### 6. 传播/讨论欲（/10）

- 标题有"站队空间"（"最危险的""真正稀缺"）：+2
- 标题带具体可转述结论（数字/对比/反常识）：+3
- 标题触达痛点但给出方向（"以后怎么办"）：+1
- 标题让读者想"这说的是我"：+2
- 标题让读者想转给同事/朋友：+2

#### 7. 语言锋利度与节奏（/5）

- 有强动词钩子（淘汰/做对/重做/颠覆/做错）：+2
- 对仗工整（4字 vs 4字，5字 vs 5字）：+1
- 押韵/金句潜力：+1
- 短句利落（无"在""的"等冗余）：+1

### Step 4：扣分项

**逐条检查标题是否触发扣分项**（独立于七维，可叠加）：

| 扣分症状 | 扣分 | 检查方法 |
|---|---|---|
| 直接给结论但没有问题感 | -5 ~ -15 | 标题是"X 是 Y" 的判断句吗？ |
| 抽象词太多，缺少具体对象 | -5 ~ -20 | 抽象名词（本质/价值/范式/趋势/复杂性）> 2 个？ |
| 太像行业评论/AI 泛观点 | -5 ~ -15 | 标题去掉"Claude Code"后还成立吗？成立 = 泛观点 |
| 读者无法判断和自己有什么关系 | -10 ~ -20 | "我们"代词 / 宽泛关键词（"未来""AI"）没具体锚点？|
| 虽然结构完整但点击欲弱 | -5 ~ -15 | 完整"我以为A结果B"但缺热词/身份词？|
| 字数 > 28 且信息密度不够 | -5 ~ -10 | 字数 28+ 但钩子在前 22 字内已展示？|
| 关键词太宽泛，没有具体锚点 | -5 ~ -15 | "AI 时代""未来趋势""大模型""创业" 缺具体工具/场景？|

## 模式 B：全文打分（资深主编版）

**评整篇文章的传播力、阅读体验、认知价值**——不评观点对错，只评"作为公众号内容"的表现。

### 适用场景

- 写完正文后看整体内容质量
- 发布前最终质检（封面做完之后、模式 A 之外补一次"内容本体"评估）
- 写完一批文章横向对比哪些能爆
- 复盘旧文为什么没火

### 评分框架（八维 100 分）

| # | 维度 | 分值 | 核心问题 |
|---|---|---|---|
| 1 | **标题** | 15 | 身份代入/热词/反直觉/问题感/点击欲 |
| 2 | **开头** | 10 | 3 段内抓住读者？是否废话/铺垫过长？ |
| 3 | **案例质量** | 15 | 真实？具体？有冲突有反转？能独立成故事？ |
| 4 | **认知增量** | 20 | 新框架？新解释？新视角？还是常识重述？ |
| 5 | **升维能力** | 10 | 是否完成"现象→规律→方法论"三段抽象？ |
| 6 | **方法论** | 10 | 给出可复用框架/判断框架？还是只是感想？ |
| 7 | **金句密度** | 10 | 可高亮的句子有几句（按阈值换算） |
| 8 | **转发潜力** | 10 | 转给同事/朋友/发群讨论的动力？ |

### 工作流

**Step 1：接收用户输入**

用户需要提供：
- 【正文】（必填，markdown 全文）
- 【标题】（必填，文章标题）
- 【目标读者】（可选，但强烈建议填——影响认知增量、案例质量的判断锚点）

**Step 2：八维打分**

每个维度**必须给具体扣分理由**。

#### 1. 标题（/15）
身份代入（"我"/"你"/具体身份词）+ 热词（Claude Code/vibe coding 等）+ 反直觉（"我以为 A，结果 B"）+ 问题感（"为什么 X 但 Y"）+ 点击欲（用户想知道"凭什么这么说"）。
**字数扣分**：标题长度按模式 A 的 Step 0 扣分表处理，**不挤进此维度**——见总分前的"零、标题长度预检"。

#### 2. 开头（/10）
3 段内抓住读者（共鸣/震惊/反差）；建立共鸣（"我也是"）；引出问题（"那什么是 X？"）。
**扣分信号**：❷ 第一句过渡平淡；铺垫超过 3 段才进入冲突。

#### 3. 案例质量（/15）
- 真实（不是"假设"举例）
- 具体（有产品名/数据/时间）
- 有冲突（"我以为 X → 验证出 Y"）
- 有反转（最终结论和初始目标不一样）
- **关键判断**：案例是"观点举例"（一句话带过）还是"独立成故事"（能讲 200 字有画面）？

**扣分信号**：案例只是"我做了 X → 验证出 Y"的平铺叙述，没有对话/现场/用户反应。

#### 4. 认知增量（/20）
- 新框架（"X 不是 Y，是 Z"）
- 新解释（用新锚点重述已知概念）
- 新视角（同一个事实从另一个角度看）
**扣分信号**：核心洞察和行业已知概念（Lean Startup / Build-Measure-Learn / MVP）完全一样，只是换了 AI 锚点。
**加分信号**：原创洞察 + 给了新锚点。

#### 5. 升维能力（/10）
三层抽象：
- 现象（具体事件）
- 规律（重复出现的模式）
- 方法论（可复用的判断/观察框架）
**扣分信号**：止步于操作建议，缺"时代判断"那一层。
**加分信号**：从方法论再升一层，给出"X 时代的产品经理/创业者的根本性转变"那种封门级终局判断。

#### 6. 方法论（/10）
- 判断框架（"以后看到 X，先问 Y"）
- 观察框架（"判断一个 Demo 价值的 3 个维度"）
- 可复用思维模型
**扣分信号**：方法论单一（只有一个"先问 X"框架），没有反向框架或边界条件。

#### 7. 金句密度（/10）
按"值得高亮的话有几句"统计，**用下面阈值换算**：

| 高亮句数 | 得分 |
|---|---|
| 0-2 句 | 3 分以下 |
| 3-5 句 | 6 分左右 |
| 6-8 句 | 8 分以上 |
| 8 句以上 | 10 分（封顶） |

"值得高亮"的判断标准：
- 反直觉 + 短句 + 有视觉感
- 单独成段也成立
- 截图发朋友圈不丢脸

**必须列出所有高亮句子**，按传播优先级排。

#### 8. 转发潜力（/10）
- 转给同事（"我们是不是也这样"）
- 转给朋友（"做产品的朋友看看"）
- 发群讨论（能立刻站队/吵架）
**扣分信号**：缺刺痛点，只是"温和说服"——读者看完觉得"有道理"但不转发。
**加分信号**：有一句"刺痛感"或"反共识判断"——读者坐不住要转。

### Step 3：流量潜力评级

| 总分 | 评级 |
|---|---|
| 85+ | 爆款潜力 |
| 75-84 | 高潜力 |
| 60-74 | 中等 |
| 45-59 | 偏弱 |
| <45 | 很弱 |

**关键判断逻辑**：
- 内容分和标题分是乘积关系，不是加和
- 标题 90+ + 内容 60+ → 标题撑住高阅读量
- 标题 60+ + 内容 90+ → 标题是天花板，需要换标题
- 标题 90+ + 内容 90+ → 爆款
- 标题 60+ + 内容 60+ → 平庸

## 关键原则：结构完整性 ≠ 真实点击潜力

**这是本 skill 最核心的校准**。区分两者：

| 维度 | 结构完整性 | 真实点击潜力 |
|---|---|---|
| 关注 | 标题"写得好不好"（弧线完整/对仗/时序） | 标题"能不能被点开"（热词/身份/1秒理解/具体承诺）|
| 加分项 | "我以为 A，结果 B" 完整弧线 | 命中具体身份词/工具词/场景词 |
| | 完美对仗 | 强动词钩子（淘汰/重做/做错）|
| | 完整时序（半年后/后来才发现） | 短句利落（15-20 字）|
| | 强反转（不是 A 是 B） | 信息承诺明确（具体复盘/数字/判断）|
| 减分项 | 缺弧线/缺时序/缺反转 | 缺热词/缺身份词/抽象词太多/字数超 28 |

**绝对禁止的隐性偏差**：
1. ❌ 不要因为标题有"我以为/后来发现/帮我/时序"就自动给高分
2. ❌ 不要因为标题"结构工整"就判定"高点击"
3. ❌ 不要把"抽象高级"误判成"高点击"
4. ❌ 不要给"金句潜力"加隐藏分——金句在正文里，标题是钩子不是金句
5. ✅ 结构完整性只能是**加分项**（最高 +5），不能替代**热词/身份代入/1秒理解/具体信息承诺**（最高 -20 扣分项）

**典型陷阱**（要记住，下次自动避坑）：

❌ 高结构分低点击的标题（实际案例）：

- **旧版评分："Claude 帮我做产品，我才发现是淘汰错误想法"**
  - 旧版结构分：95/100（"我以为A，结果B"完整弧线 + "在帮我"亲切感 + "后来发现"时序）
  - **新版点击潜力：38/100（偏弱）**
    - 抽象词"淘汰错误想法"：-10
    - 去掉"Claude"后仍成立（泛观点）：-8
    - 关键词太宽泛：-8
    - 缺具体数字/动作锚点
  - **新规则驱动发现了 v18a："Claude Code 半年，92% 都在做错" → 84/100（高潜力）**
    - 用正文真实数字 92% 替代抽象的"淘汰错误想法"
    - 数字反差（92% 错 vs 8% 对）远强于结构完整度

- "Claude Code 用半年我才发现：它做的是产品实验"（结构 96，但"产品实验"是产品圈行话，非产品岗读者 1 秒内不完全理解）

✅ 低结构分高点击的标题：

- "Claude Code 半年，帮我省了 100 个小时"（结构可能 70，但"省 100 小时"是具体数字钩子，点击更高）
- "程序员用 Claude Code 的第 30 天，我辞职了"（结构 75，但"辞职"是强冲突 + "程序员"是身份词，点击远高于结构 96 的标题）

**核心教训**：旧版评分系统性高估结构完整标题的点击潜力。当发现标题"结构分高但没有具体数字/强动词/身份锚点"时，用新规则重新打分——往往比预期低 40-60 分。

**结论**：评"标题是否会被点开"——不要评"标题是否写得工整"。

### 模式 B 陷阱

1. **不要评观点对错**——用户明确说不要。即使你觉得"Demo 价值不是上线"是错的，也不要扣分。
2. **不要混淆模式 A 和 B**——模式 A 看到正文也不评；模式 B 必须评。
3. **不要把"金句多"等同于"内容好"**——案例空、升维空，但堆了 10 个金句，依然是中等偏弱。金句密度只是八维之一。
4. **不要用"高潜力"模糊评级**——必须明确给出"爆款/高/中/偏弱/很弱"之一，并给一句话理由。
5. **不要省略"如果阅读量不高最可能的原因"**——这是用户最想知道的反推逻辑。
6. **必须修改只能给 1 个**——给多了用户就只改一个，干脆只挑 1 个最值改的。
7. **金句数量按"反直觉+短句+视觉感"标准数**——不要把"几小时一个产品"这种"具体细节"也算金句。判断：截图发朋友圈丢不丢脸？丢脸就不算。

## 已知陷阱（模式 A）

1. **外链封面 vision 经常失败**——优先用 file:// 路径 + mcp_browser_vision
2. **"AI 没有替代程序员"这种标题**——目标读者代入感弱（没说"你"也没说"程序员"），属于"看起来在讨论程序员但你不知道是不是写给你的"。要建议加"我们""你"或具体身份词
3. **"X只是把Y重新分层"句式**——认知冲突分高（反转），但"重新分层"是抽象词，扣信息承诺分
4. **封面用纯英文大字**——符合公众号封面规范（用户在 wechat-post 偏好里固化了"纯英文、超大白字、黑色粗描边"），不要扣分
5. **不要给"完整文章评估"**——用户可能想测的是单看标题/封面吸引力，正文好不好是另外的事
6. **副标题在手机信息流几乎不可见**——副标题字号约 12-14px，绝大多数人看不到。主标题必须自己完成"话题+反转"双重任务，不能依赖副标题承载核心冲突。
7. **"AI 时代"开头的标题自动降权**——在程序员/产品经理/AI创业者群体里，"AI 时代 + 大判断"型标题被识别为 AI 观点文，0.3 秒被跳过。保留"AI 时代"必须把它推到"被讽刺/被解构"的位置（如反问句或反转句里）。
8. **直接给结论型标题（如"被严重低估了"）缺问题感**——读者看完不知道"低估了什么/我之前怎么以为的"，认知冲突分为 0。必须改成问题式或反差式结构（"我以为A结果B" / "为什么X但Y" / "不是X，是Y"）。
9. **第一档标题格式（"为什么...但..."）天然高 20+ 分**——"为什么"开头 = 强问题感，"很多 Demo 没人用" = 强冲突 + 身份词落地，"但我还在做" = 强反转。这是基准线，任何低于这个结构的标题都要扣分。

## 全局陷阱

1. **模式 A vs B 选错**——用户只说"打分"时默认走模式 A；用户明确说"全文打分 / 正文打分 / 看完整文章"才走模式 B。模式 A 看到正文不要顺手评，模式 B 评了就要评全套八维。
2. **两种模式不要混用框架**——模式 A 七维是"标题+封面在信息流的吸引力"（不打分内容）；模式 B 八维是"全文传播力"（不评观点对错）。两套分数不能放在同一份报告里。
3. **存档位置**——打分结果存到 `~/.hermes/research/score_YYYYMMDD_HHMMSS.md`（模式 A）或 `score_full_YYYYMMDD_HHMMSS.md`（模式 B），便于复盘历史评分。
4. **迭代使用**——用户说"再打分"/"改一版"时，调取上次打分结果对照，看分数升降。
5. **标题长度预检两种模式都要走**——Step 0 字数扣分表是统一的（15-20 字最佳，过短过长都扣），不挤进七维也不挤进八维，独立显示在"零、标题长度预检"区。**不要再用老的"22 字以上扣 1-2 分"松散规则**。
6. **数字单位一致**——"我以为 X，结果 Y"型标题在 Step 0 算字数时，"我"、"以为"、"结果"都是 1 字；阿拉伯数字"92%"= 1 字；英文单词按 1 字符 = 1 字；英文标题按 1 词 = 1.5 字折算。
7. **标题-正文一致性是隐性大坑**——Step 0.5 预检常被忽略，但"读完觉得被骗了"是低传播率最隐蔽的根因。任何标题"用工具/具体经验"但正文"讲方法论/趋势"的组合都要 -3 起步。改法：把标题视角对齐到正文实际讲的视角，不要试图"两边兼顾"。
8. **拗口是隐性杀手**——Step 0.6 预检的最严重症状是 2 对以上引号 + 主语跳跃 + "X 不如 Y" 双方都是生造短语。这种标题读者需要重读，0.3 秒内直接划走。"X 不如 Y" 结构的 X 和 Y 必须是大众词（快/慢/贵/便宜/稳/玄等）。
9. **结构完整性 ≠ 真实点击潜力**——不要因为标题有"我以为/后来发现/帮我/时序"就给高分。结构分只是加分项（最多 +5），不能替代热词/身份词/1秒理解/具体信息承诺（最高 -20 扣分）。"结构 96 但点击弱" 的典型是"我以为 Claude 在做产品，后来发现是淘汰错误想法"——结构完整但"淘汰错误想法"对很多读者抽象。
10. **去掉"Claude Code" 后还成立的标题是泛观点**——"AI 时代产品方法论彻底变了" 去掉 Claude Code 完全成立 = 行业泛观点（扣 5-15）。好的标题必须"和 Claude Code/具体工具/具体身份强绑定"。
11. **短期流量高 ≠ 标题好**——新加"真实性 15 分"和"账号风格 10 分"就是为了防止"短期流量自杀长期品牌"。一个标题可能七维 95 但真实性 0（标题党）、账号风格 0（数字党/震惊体）——综合分 95+0+0=95（<100 综合不推荐）。**综合分 < 100 的标题慎用，< 80 不推荐**。
12. **数字有出处是底线**——"92% 都在做错"如果正文有"92% 评估准确率 + 11% 需求命中"的真实数据，加真实性满分；如果是凭空编的"92% 程序员都这样"，真实性 0。"行业数据"和"作者亲历"差很多——前者读者会想"数据从哪来"，后者读者会想"我也这样"。
