# 公众号内容打分

两种模式，按用户问的来：

| 模式 | 评估对象 | 何时用 | 框架 |
|---|---|---|---|
| **A. 标题+封面打分** | 只评标题+封面在信息流的吸引力 | 写完正文、做封面之前；封面做完、发布前；历史复盘 | 七维 100 分制（见下文） |
| **B. 全文打分** | 评整篇文章的传播力、阅读体验、认知价值 | 写完正文后看整体内容质量；发布前最终质检；写完一批文章对比 | 资深主编版八维 100 分制（见下文） |

**关键原则**：
- 模式 A 不评正文（用户明确说不要）
- 模式 B 评全文但**不评观点对错**（用户明确说不要）——只评"作为公众号内容的传播力"
- 用户没说哪种时，默认按"标题+封面打分"（模式 A）

---

## 模式 A：标题 + 封面流量潜力打分

只评估**标题 + 封面**在公众号信息流里的吸引力，不评正文质量。

## 适用场景

- 写完文章准备发之前，给标题和封面打分
- 看别人文章时复盘为什么火/为什么没人看
- 同一篇文章尝试多个标题时挑最优

## 评分框架（总分 100）

| 维度 | 分值 | 核心问题 |
|---|---|---|
| 1. 目标读者代入感 | 20 | 一眼觉得"这是在说我"？ |
| 2. 热词和时效性 | 15 | 是否接入已有讨论场？ |
| 3. 认知冲突/反直觉 | 20 | 是否有"我以为A，结果B"的反差？ |
| 4. 信息承诺清晰度 | 15 | 点进去会获得什么具体东西？ |
| 5. 封面视觉焦点 | 15 | 缩略图里有没有视觉中心？ |
| 6. 标题封面一致性 | 10 | 封面是否强化标题的核心冲突？ |
| 7. 转发欲/讨论欲 | 5 | 会不会想转给同事/站队？ |

## 工作流

### Step 0：标题长度预检（必走，独立于七维）

公众号信息流里，标题的物理显示宽度有限。
- 手机端单行约显示 15-20 字（中文按 1 字算，英文按 1 字符算 1 字，数字按 1 字算）
- 超过 22 字必被截断，钩子失效
- 短于 11 字，身份词/反转/冲突塞不进去
- **15-20 字是最佳区间**（既能在缩略图完整显示，又能塞下"身份词 + 反转"）

**字数扣分表**（从总分里直接扣，不挤进七维）：

| 标题字数 | 扣分 | 原因 |
|---|---|---|
| 15-20 字 | 0 | 最佳区间，单行完整显示+信息密度够 |
| 13-14 或 21-22 | -1 | 边界外，可接受但欠优 |
| 11-12 或 23-24 | -2 | 明显偏离，信息密度或显示完整度不足 |
| 9-10 或 25-26 | -3 | 严重偏离，要么太空要么必截断 |
| <9 或 ≥27 | -4 | 极短（丢失冲突/身份词）或必截断（前缀钩子丢失） |

**注**：
- "字" = 中文字符、英文字母按 1 字符、数字按 1 字符、英文空格不计入
- 例外：纯英文标题（无中文）按 1 词 = 1.5 字折算

### Step 0.5：标题-正文一致性预检（必走，独立于七维）

**这是公众号最隐蔽也最致命的扣分项**——读完落差让读者觉得"被骗了"，直接打击转发欲（往往是低传播率的根因）。

**判断方法**：
1. 标题里的核心关键词（如 "Claude Code"）点出来的是什么视角？—— 工具视角 / 方法论视角 / 趋势视角
2. 正文实际在讲什么视角？—— 用一句话总结正文的真正核心观点
3. **两者是否同一视角？**

**扣分表**（从总分里直接扣，不挤进七维）：

| 错位程度 | 扣分 | 典型症状 |
|---|---|---|
| 完全一致 | 0 | 标题承诺的"会得到什么"和正文实际给的"是什么"对得上 |
| 视角错位 | -3 | 标题说"工具"，正文讲"方法论"；或标题说"个人经验"，正文讲"行业判断"（典型：工程师点进"Claude Code 半年"标题，结果读到产品哲学） |
| 范围错位 | -5 | 标题限定一个具体场景（如"做八字 Skill"），正文泛化到整个产品方法论（读者期望八卦/具体复盘，结果读到一篇大而泛的方法论文） |
| 承诺完全不符 | -8 | 标题说"A 体验"，正文讲"B 主题"（这种基本是不小心的写作事故） |

**典型错位举例**（要记住，下次自动避坑）：

❌ 标题："Claude Code 用半年我才发现：它做的是产品实验"
✅ 正文实际：讲的是 AI 时代产品方法论（"做对→做快"）
❌ 错位：标题锁定工具视角（"它做的是 X"），正文实际是方法论视角
❌ 后果：工程师点进来看 Claude Code 技巧 → 读完发现是产品哲学 → 觉得"被骗了" → 不转发
**应改成**：
- 改成方法论视角："Claude Code 用半年我才发现：做产品方法论变了" / "Claude Code 半年后：AI 产品方法论从'做对'变成'做快'" / "Claude Code 用半年：让我重新理解'做对'和'做快'"
- 或者改成工具视角的真正具体经验（不写方法论）

**修法检查清单**：
- 改完标题后，把标题关键词列出来
- 再用一句话总结正文核心观点
- 两者视角必须一致（要么都"工具"，要么都"方法论"，要么都"个人经验"）
- 不能一个限定"工具"另一个讲"方法论"

### Step 0.6：标题拗口度预检（必走，独立于七维）

**拗口 = 重复读 = 信息流划走**。公众号信息流里读者读标题 0.3 秒，**任何需要"重新组句才能懂"的标题就是拗口**。

**判断方法（逐条检查）**：

1. **双引号/书名号数量**：标题里出现 2 个及以上"X"或《X》→ 大概率拗口（每出现一组 +1 拗口分）
2. **主语跳跃**：标题前半部分在讲 A，后半部分突然转 B，中间没有连接词 → 拗口
3. **"X 不如 Y" 结构**："X"和"Y"必须都是读者已知的明确概念，否则读者会卡在"X是什么"——拗口
4. **嵌套结构**："A，B，C，D"超过 3 个短句、或"X，Y"出现 2 次以上 → 拗口
5. **绕口令测试**：标题一口气读完是否要中途换气 2 次以上？要换气 2 次 = 拗口

**拗口扣分表**（从总分里直接扣，可叠加，最多 -8）：

| 拗口症状 | 扣分 | 典型案例 |
|---|---|---|
| 0 个拗口症状 | 0 | 流畅，一口气读完 |
| 1 个轻微症状（如 1 对引号 / 1 次主语跳跃） | -2 | 还能读，但不顺 |
| 2 个症状（如 2 对引号 + 1 次主语跳跃） | -4 | 明显拗口 |
| 3 个症状（如 2 对引号 + 主语跳跃 + "X 不如 Y" 都陌生） | -6 | 必须重读 |
| 4 个及以上 | -8 | 灾难级拗口 |

**改法原则**：
- **去掉引号**：除非你引的是大众熟悉的金句（"996" "35 岁"），否则一律去掉
- **主语统一**：标题全程围绕同一个主语，不要从 A 跳到 B
- **"X 不如 Y" 的 X 和 Y 必须是大众词**：X=快、便宜、稳（已知概念）；Y=慢、贵、玄（已知概念）。如果 X 或 Y 是生造词，必须改写成大众词
- **短句重排**：如果标题有 3 个以上短句，砍到 2 个短句（逗号最多 1 个）

**改法举例**：

❌ `Claude Code 半年，我才看懂"做对"不如"做快"`
拗口：2 对引号 + 主语跳跃（Claude Code → 做对 → 做快）+ "X 不如 Y" 双方都是生造短语
✅ 改成 `Claude Code 半年，我才看懂一件事：做对不如做快`（去掉引号，主语统一为"一件事"，X/Y 简化为大众词）
✅ 或 `Claude Code 半年，让我重新理解了一件事`（彻底规避"X 不如 Y"结构）
✅ 或 `Claude Code 半年，我才看懂"做对"是个错觉`（保留 1 对引号，但 X 是大众词，Y=错觉也是大众词）

### Step 1：接收用户输入

用户需要提供：
- 【目标读者】（必填，例：程序员 / AI工程师 / OpenClaw用户）
- 【标题】（必填）
- 【封面】（必填，图片路径或URL）
- 【已知参考数据】（可选，历史阅读/点赞/分享数据）

如果只提供标题没有封面：明确告知"封面维度无法评估"，其余六维照打。

### Step 2：评估封面

封面评估**必须调用 vision 工具实际看图**，不要凭空判断。

**方法 A（推荐）**：用 `mcp_browser_navigate` 打开 file:// 本地路径，再 `mcp_browser_vision` 出图分析。
**方法 B**：`mcp_vision_analyze` 直接传本地路径（外链URL常失败）。

**评估清单**（缩略图视角，假设手机端约 200x150 像素）：

| 检查项 | 红旗信号 |
|---|---|
| 视觉中心 | 没有明确主体，或有2个以上并列主体 |
| 主文字字号 | <画面15%面积，或多行小字 |
| 元素数量 | 主体+文字+装饰>4个独立元素 |
| 配色对比 | 主文字与背景亮度差<40% |
| 缩略图可读性 | 缩到200x150后主体仍然清晰 |
| 视觉层级 | 没有"主标题文字>主体图像>背景"三段式 |

### Step 3：七维打分

每个维度**必须给具体扣分理由**，不能直接给总分。

#### 1. 目标读者代入感（/20）

**加分词**（命中任意+3~5分）：程序员 / 软件工程 / vibe coding / Claude Code / Cursor / Agent / OpenClaw / AI创业 / 副业 / 35岁 / 转型 / 失业
**减分信号**：标题里完全没有身份词/场景词（"我们""你"等代词不算）

#### 2. 热词和时效性（/15）

**加分**：标题包含最近3个月目标读者高频讨论的关键词（Claude Code / Cursor / Codex / Agent / Vibe Coding / Prompt / 上下文工程 / Skills 等）
**减分**：完全泛AI观点（"AI时代来临"），无具体产品/技术/事件挂钩

#### 3. 认知冲突/反直觉（/20）

**高分量表达**：
- "我以为A，结果B"（"以为…其实…"）
- 反问句（"为什么…？"）
- 反转（"不是X，是Y"）
- 数字反差（"92%技术评估准确率，只有11%需求命中"）
- 极值词（"最危险的""真正稀缺"）

**减分**：标题是"AI时代要X""程序员应该Y"这种无冲突的判断句

#### 4. 信息承诺清晰度（/15）

**加分**：读者能预知文章会给出"具体复盘/具体判断/具体数字"
**减分**：
- 标题像口号（"未来已来""新范式"）
- 标题太抽象（"复杂性""价值"等抽象名词堆砌）
- 标题没有承诺任何"读者会得到的东西"

#### 5. 封面视觉焦点（/15）

按 Step 2 评估清单的实际观察打分。

| 现象 | 分数 |
|---|---|
| 缩略图一眼看清主体+主文字，视觉中心强 | 12-15 |
| 主体清晰但主文字偏小，或主文字够大但主体弱 | 8-11 |
| 元素多/散/层级混乱 | 4-7 |
| 完全是装饰图无文字，或文字过小不可读 | 0-3 |

#### 6. 标题封面一致性（/10）

- 封面文字与标题核心冲突点一致：+3
- 封面主体形象能强化标题场景：+3
- 视觉风格调性匹配（技术文配冷色，生活文配暖色）：+2
- 1秒内能让读者猜出"这篇在讲什么"：+2

#### 7. 转发欲/讨论欲（/5）

- 标题有"站队空间"（"最危险的""真正稀缺"）：+2
- 标题带具体可转述结论（数字/对比/反常识）：+2
- 标题触达痛点但给出方向（"以后怎么办"）：+1

### Step 4：总评输出格式

**严格按以下格式输出**，便于复制到文档：

```
【零、标题长度预检】
- 字数：X 字
- 区间：15-20 字最佳 / 偏离则按扣分表
- 扣分：-X（或 0）

【零.5、标题-正文一致性预检】
- 标题视角：（工具视角 / 方法论视角 / 趋势视角 / ...）
- 正文视角：（用一句话总结）
- 是否一致：是 / 视角错位 / 范围错位 / 承诺不符
- 扣分：-X（或 0）

【零.6、标题拗口度预检】
- 拗口症状：列出（如"2 对引号 + 1 次主语跳跃"）
- 拗口分：X 个症状

【四、最大问题】
```

## 输出风格

- 评估要**锐利**，敢于直接说"这个标题不行"
- 改写标题要**具体到能直接用**，不是泛泛的"建议加个反差"
- 封面问题要**指出具体动作**（"把主文字放大2倍"），不是说"建议优化"
- 不评正文质量，被问到正文就说"超出本次评估范围"

## 模式 B：全文打分（资深主编版）

**评整篇文章的传播力、阅读体验、认知价值**——不评观点对错，只评"作为公众号内容"的表现。

### 适用场景

- 写完正文后看整体内容质量
- 发布前最终质检（封面做完之后、模式 A 之外补一次"内容本体"评估）
- 写完一批文章横向对比哪些能爆
- 复盘旧文为什么没火

### 评分框架（八维 100 分）

| # | 维度 | 分值 | 核心问题 |
|---|---|---|---|
| 1 | **标题** | 15 | 身份代入/热词/反直觉/问题感/点击欲 |
| 2 | **开头** | 10 | 3 段内抓住读者？是否废话/铺垫过长？ |
| 3 | **案例质量** | 15 | 真实？具体？有冲突有反转？能独立成故事？ |
| 4 | **认知增量** | 20 | 新框架？新解释？新视角？还是常识重述？ |
| 5 | **升维能力** | 10 | 是否完成"现象→规律→方法论"三段抽象？ |
| 6 | **方法论** | 10 | 给出可复用框架/判断框架？还是只是感想？ |
| 7 | **金句密度** | 10 | 可高亮的句子有几句（按阈值换算） |
| 8 | **转发潜力** | 10 | 转给同事/朋友/发群讨论的动力？ |

### 工作流

**Step 1：接收用户输入**

用户需要提供：
- 【正文】（必填，markdown 全文）
- 【标题】（必填，文章标题）
- 【目标读者】（可选，但强烈建议填——影响认知增量、案例质量的判断锚点）

**Step 2：八维打分**

每个维度**必须给具体扣分理由**。

#### 1. 标题（/15）
身份代入（"我"/"你"/具体身份词）+ 热词（Claude Code/vibe coding 等）+ 反直觉（"我以为 A，结果 B"）+ 问题感（"为什么 X 但 Y"）+ 点击欲（用户想知道"凭什么这么说"）。
**字数扣分**：标题长度按模式 A 的 Step 0 扣分表处理，**不挤进此维度**——见总分前的"零、标题长度预检"。

#### 2. 开头（/10）
3 段内抓住读者（共鸣/震惊/反差）；建立共鸣（"我也是"）；引出问题（"那什么是 X？"）。
**扣分信号**：❷ 第一句过渡平淡；铺垫超过 3 段才进入冲突。

#### 3. 案例质量（/15）
- 真实（不是"假设"举例）
- 具体（有产品名/数据/时间）
- 有冲突（"我以为 X → 验证出 Y"）
- 有反转（最终结论和初始目标不一样）
- **关键判断**：案例是"观点举例"（一句话带过）还是"独立成故事"（能讲 200 字有画面）？

**扣分信号**：案例只是"我做了 X → 验证出 Y"的平铺叙述，没有对话/现场/用户反应。

#### 4. 认知增量（/20）
- 新框架（"X 不是 Y，是 Z"）
- 新解释（用新锚点重述已知概念）
- 新视角（同一个事实从另一个角度看）
**扣分信号**：核心洞察和行业已知概念（Lean Startup / Build-Measure-Learn / MVP）完全一样，只是换了 AI 锚点。
**加分信号**：原创洞察 + 给了新锚点。

#### 5. 升维能力（/10）
三层抽象：
- 现象（具体事件）
- 规律（重复出现的模式）
- 方法论（可复用的判断/观察框架）
**扣分信号**：止步于操作建议，缺"时代判断"那一层。
**加分信号**：从方法论再升一层，给出"X 时代的产品经理/创业者的根本性转变"那种封门级终局判断。

#### 6. 方法论（/10）
- 判断框架（"以后看到 X，先问 Y"）
- 观察框架（"判断一个 Demo 价值的 3 个维度"）
- 可复用思维模型
**扣分信号**：方法论单一（只有一个"先问 X"框架），没有反向框架或边界条件。

#### 7. 金句密度（/10）
按"值得高亮的话有几句"统计，**用下面阈值换算**：

| 高亮句数 | 得分 |
|---|---|
| 0-2 句 | 3 分以下 |
| 3-5 句 | 6 分左右 |
| 6-8 句 | 8 分以上 |
| 8 句以上 | 10 分（封顶） |

"值得高亮"的判断标准：
- 反直觉 + 短句 + 有视觉感
- 单独成段也成立
- 截图发朋友圈不丢脸

**必须列出所有高亮句子**，按传播优先级排。

#### 8. 转发潜力（/10）
- 转给同事（"我们是不是也这样"）
- 转给朋友（"做产品的朋友看看"）
- 发群讨论（能立刻站队/吵架）
**扣分信号**：缺刺痛点，只是"温和说服"——读者看完觉得"有道理"但不转发。
**加分信号**：有一句"刺痛感"或"反共识判断"——读者坐不住要转。

### Step 3：流量潜力评级

| 总分 | 评级 |
|---|---|
| 85+ | 爆款潜力 |
| 75-84 | 高潜力 |
| 60-74 | 中等 |
| 45-59 | 偏弱 |
| <45 | 很弱 |

**关键判断逻辑**：
- 内容分和标题分是乘积关系，不是加和
- 标题 90+ + 内容 60+ → 标题撑住高阅读量
- 标题 60+ + 内容 90+ → 标题是天花板，需要换标题
- 标题 90+ + 内容 90+ → 爆款
- 标题 60+ + 内容 60+ → 平庸

### 输出风格

- 评估要**像资深主编**：敢说"78 分，不是爆款上限"
- 不给模糊分数，每个维度都给具体扣分理由
- **必须修改只能选 1 个最值得改的**——不要列一堆让用户选择困难
- **金句必须全部列出来**——这才是用户能从你这拿走的核心资产
- 转发潜力要给"具体是什么场景"——"转给同事看"还是"发群讨论"必须分清

### 模式 B 陷阱

1. **不要评观点对错**——用户明确说不要。即使你觉得"Demo 价值不是上线"是错的，也不要扣分。
2. **不要混淆模式 A 和 B**——模式 A 看到正文也不评；模式 B 必须评。
3. **不要把"金句多"等同于"内容好"**——案例空、升维空，但堆了 10 个金句，依然是中等偏弱。金句密度只是八维之一。
4. **不要用"高潜力"模糊评级**——必须明确给出"爆款/高/中/偏弱/很弱"之一，并给一句话理由。
5. **不要省略"如果阅读量不高最可能的原因"**——这是用户最想知道的反推逻辑。
6. **必须修改只能给 1 个**——给多了用户就只改一个，干脆只挑 1 个最值改的。
7. **金句数量按"反直觉+短句+视觉感"标准数**——不要把"几小时一个产品"这种"具体细节"也算金句。判断：截图发朋友圈丢不丢脸？丢脸就不算。

## 已知陷阱（模式 A）

1. **外链封面 vision 经常失败**——优先用 file:// 路径 + mcp_browser_vision
2. **"AI 没有替代程序员"这种标题**——目标读者代入感弱（没说"你"也没说"程序员"），属于"看起来在讨论程序员但你不知道是不是写给你的"。要建议加"我们""你"或具体身份词
3. **"X只是把Y重新分层"句式**——认知冲突分高（反转），但"重新分层"是抽象词，扣信息承诺分
4. **封面用纯英文大字**——符合公众号封面规范（用户在 wechat-post 偏好里固化了"纯英文、超大白字、黑色粗描边"），不要扣分
5. **不要给"完整文章评估"**——用户可能想测的是单看标题/封面吸引力，正文好不好是另外的事
6. **副标题在手机信息流几乎不可见**——副标题字号约 12-14px，绝大多数人看不到。主标题必须自己完成"话题+反转"双重任务，不能依赖副标题承载核心冲突。
7. **"AI 时代"开头的标题自动降权**——在程序员/产品经理/AI创业者群体里，"AI 时代 + 大判断"型标题被识别为 AI 观点文，0.3 秒被跳过。保留"AI 时代"必须把它推到"被讽刺/被解构"的位置（如反问句或反转句里）。
8. **直接给结论型标题（如"被严重低估了"）缺问题感**——读者看完不知道"低估了什么/我之前怎么以为的"，认知冲突分为 0。必须改成问题式或反差式结构（"我以为A结果B" / "为什么X但Y" / "不是X，是Y"）。
9. **第一档标题格式（"为什么...但..."）天然高 20+ 分**——"为什么"开头 = 强问题感，"很多 Demo 没人用" = 强冲突 + 身份词落地，"但我还在做" = 强反转。这是基准线，任何低于这个结构的标题都要扣分。

## 全局陷阱

1. **模式 A vs B 选错**——用户只说"打分"时默认走模式 A；用户明确说"全文打分 / 正文打分 / 看完整文章"才走模式 B。模式 A 看到正文不要顺手评，模式 B 评了就要评全套八维。
2. **两种模式不要混用框架**——模式 A 七维是"标题+封面在信息流的吸引力"（不打分内容）；模式 B 八维是"全文传播力"（不评观点对错）。两套分数不能放在同一份报告里。
3. **存档位置**——打分结果存到 `~/.hermes/research/score_YYYYMMDD_HHMMSS.md`（模式 A）或 `score_full_YYYYMMDD_HHMMSS.md`（模式 B），便于复盘历史评分。
4. **迭代使用**——用户说"再打分"/"改一版"时，调取上次打分结果对照，看分数升降。
5. **标题长度预检两种模式都要走**——Step 0 字数扣分表是统一的（15-20 字最佳，过短过长都扣），不挤进七维也不挤进八维，独立显示在"零、标题长度预检"区。**不要再用老的"22 字以上扣 1-2 分"松散规则**。
6. **数字单位一致**——"我以为 X，结果 Y"型标题在 Step 0 算字数时，"我"、"以为"、"结果"都是 1 字；阿拉伯数字"92%"= 1 字；英文单词按 1 字符 = 1 字；英文标题按 1 词 = 1.5 字折算。
7. **标题-正文一致性是隐性大坑**——Step 0.5 预检常被忽略，但"读完觉得被骗了"是低传播率最隐蔽的根因。任何标题"用工具/具体经验"但正文"讲方法论/趋势"的组合都要 -3 起步。改法：把标题视角对齐到正文实际讲的视角，不要试图"两边兼顾"。
8. **拗口是隐性杀手**——Step 0.6 预检的最严重症状是 2 对以上引号 + 主语跳跃 + "X 不如 Y" 双方都是生造短语。这种标题读者需要重读，0.3 秒内直接划走。"X 不如 Y" 结构的 X 和 Y 必须是大众词（快/慢/贵/便宜/稳/玄等）。
